## 1. 서비스 개요 (Service)

> **제1원칙.** 송금 절차에서 LLM을 배제하여 할루시네이션(Hallucination)에 따른 금융 사고를 원천 차단한다.
> 의도 파싱(LLM)은 자연어만 담당하고, 금액과 실행의 판정은 전부 결정적 코드(3중 게이트)가 수행한다.
> 이 명세서는 기능 나열이 아니라 통제(Control) 명세이다. 각 기능이 잘못될 때 어떻게 멈추는지, 책임은 누구에게 있는지를 함께 규정한다.

### 1.1. v1에서 v2로의 확장 (같은 엔진, 두 입구)

- **v1(근로자)**: 활성 계좌를 전제로, 급여 트리거 자동 송금(Step 1)과 사채 대환 가심사(Step 2)를 제공한다. 본 v2에서 무수정 보존한다.
- **v2(유학생 추가)**: JB가 깐 가계좌와 한도계좌와 심사엔진 위에서, 유학생(D-2)을 동일 엔진으로 흡수한다. 등록금 외화 송금, 한도계좌 해제 코치, 재학 중 신용 형성, 졸업 전환 가심사를 제공한다.
- **핵심 설계**: 근로자와 유학생은 **동일한 위임장과 동일한 3중 게이트**를 공유한다. 유학생 등록금 송금은 기존 `run_remittance`를 `fx_pair="KRW/CNY"`로 그대로 재사용하므로, 새 송금 엔진을 추가하지 않는다. 둘째 세그먼트는 시드 데이터와 표시 레이어만 추가하여 확장했고, 셋째 세그먼트도 같은 패턴으로 확장할 수 있다.

## 2. 시스템 구성도 (Architecture)

> 그림: `docs/diagrams.md` §2(3중 게이트 구성도)를 캡처해 삽입한다.

**3중 게이트 (전부 결정적 코드):**

```
[발화 / 급여 입금 이벤트] → LLM 의도 파싱(자연어만) → 구조화된 intent
   ┌─── 3중 게이트 ───┐
   │ Gate A 위임장 검증     (전자서명, 유효기간, 철회, 범위)        │
   │ Gate B Rule 한도·조건  (FX Rule, 금액 한도)                  │
   │ Gate C 화이트리스트+AML (structuring, 신규×고액×심야, 블랙리스트) │
   └──── PASS만 통과 ────┘
   → [실행 엔진] 송금 실행, 건별 모국어 통지, 철회 버튼
   → [검증·개선] 사후 성과를 반영한 위임 임계값 조정 제안
```

**컴포넌트:**

| 레이어 | 구성요소 | 역할 |
|---|---|---|
| Frontend | React(Vite) 챗 UI, 유학생 화면, 보호자·관리 대시보드 | 모국어 대화, 위임 현황, 계좌 생애주기, 감사로그 |
| Orchestrator | FastAPI(MCP 호스트) | 3중 게이트를 강제하여 LLM의 직접 실행을 차단 |
| Agent(LLM) | 의도 파싱, 설명 생성 | 실행 권한 없음. 식별정보(PII) 비전달(토큰화) |
| Rule Engine | `rules.py` (FX, AML, 대환) | 결정적 판정 |
| **Activation Layer** | `activation.py` (한도해제 판정) | 가계좌에서 정식계좌로의 승급 가능 시점을 결정적으로 코치(자금 이동 없음, read-only) |
| Mandate Service | Gate A (`esign_hash`) | 위임장 검증 |
| Mock Bank Core | SQLite (12 테이블) | 계좌, 급여, 환율, 수취인, 사채, 대환, STR, 감사로그 |
| (옵셔널리티) | 외부 AI 비서(Claude Desktop 등) | 동일 MCP 표준에 연결 가능. 종속이 아닌 선택권 |

**멀티에이전트 구성:** 대화 Agent(LLM, 실행 권한 없음), 위임검증 Agent(Gate A), 송금리스크 Agent(Gate C, AML), 대환 Agent(Step 2 가심사 후 전북은행 회부).

> **v2 핵심.** 유학생 등록금 송금은 근로자 자동 송금과 동일한 위임장과 동일한 3중 게이트를 `fx_pair="KRW/CNY"`로 재사용한다. 활성화 레이어(`activation.py`)는 송금 실행 경로 밖의 read-only 판정 모듈로, 동결 구역(orchestrator, rules)을 건드리지 않는다.

### 2.5. 이 시스템이 하지 않는 것 (설계로 그은 경계)

> 보수적인 금융 환경에서 신뢰는 무엇을 할 수 있는가보다 무엇을 막아 두었는가에서 나온다. 아래 5개는 선택 옵션이 아니라 아키텍처 제약이다.

| # | 시스템이 하지 않는 것 | 강제 방식 |
|---|---|---|
| 1 | LLM이 송금을 실행하지 않는다 | `execute_remittance`는 오케스트레이터가 3중 게이트 PASS 이후에만 호출하며, LLM이 직접 호출할 수 없다 |
| 2 | 에이전트가 대출을 승인하지 않는다 | 최종 승인 권한은 전북은행 심사엔진에 위임하고, 에이전트는 `REFER_TO_JB_ENGINE`(회부)만 반환한다 |
| 3 | 환율과 사기를 예측하지 않는다 | FX와 AML 모두 학습 모델이 아니라, 사용자가 정한 임계조건에 따른 결정적 Rule과 스코어로 판정한다 |
| 4 | 외부 LLM에 식별정보를 보내지 않는다 | 의도 파싱과 번역만 전달하고, 거래 데이터는 토큰화한다 |
| 5 | 위임 범위 밖을 임의로 실행하지 않는다 | 화이트리스트 위반, 한도 초과, 의심 거래는 `hold_and_ask`(보류, 모국어 질문)로 처리하고, 무응답 시 자동 취소한다 |

> 단일 사고의 최대 손실은 위임 한도 내로 한정된다. 화이트리스트, 1회·월 한도, 100% 철회권으로 상한(Capping)을 건다.

## 3. 핵심 기능 명세 (Feature Specification)

> 핵심 기능 위주로 기술한다. MCP 도구가 코어이며, 굵게 표기한 도구가 3중 게이트이다.

### 3-1. 기능·통제 명세 (Control Spec)

> 정상 동작이 아니라 실패·보류 시 동작과 책임 경계를 함께 규정한다. 돈을 움직이는 시스템에서는 멈추는 법이 1급 기능이다.

#### (가) 근로자 기능 (v1, 무수정 보존)

| ID | 기능 | 단계 | 실패·보류 시 동작 | 책임 경계 |
|---|---|---|---|---|
| F-01 | 위임장 발급 | 생성 | 모국어 재확인 미완 또는 서명 누락 시 발급 거부 | 위임 내용은 `esign_hash`로 고객이 입증·동의 |
| F-02 | 급여 입금 감지 | 수집·트리거 | 트리거 미충족 시 대기(실행 안 함) | 시스템(이벤트 감지) |
| F-03 | 환율 조회·이동평균 | 수집·Gate B | 환율 데이터 60초 지연 시 거래 정지(HOLD) | 시스템 |
| F-04 | **위임장 검증 (Gate A)** | **판단** | 만료, 철회, 서명 누락, 범위 밖이면 거절·보류 | 시스템(결정적 코드) |
| F-05 | **한도·FX 조건 (Gate B)** | **판단** | 한도 초과 시 거절, FX 미달 시 대기(6시간 재평가) | 한도는 고객이 위임장에서 사전 설정 |
| F-06 | **AML 스크리닝 (Gate C)** | **판단** | 점수 40 이상 보류, 70 이상 STR 후보 큐 등록과 함께 보류, 블랙리스트 일치 시 점수와 무관하게 즉시 차단 | 시스템 탐지, STR 보고 판단은 준법감시 |
| F-07 | 송금 실행 | **행동** | 3중 게이트 PASS 이후에만 실행, 한 건이라도 실패 시 미실행 | 정상 PASS는 고객 위임, 시스템 오류는 배상한도 약관(로드맵) |
| F-08 | 건별 모국어 통지 | 생성 | 통지 실패 시 재시도와 로그 기록 | 시스템 |
| F-09 | 위임 철회 | 후속액션 | 즉시 `revoked=true`, 진행 트리거 중단 | 100% 철회권은 고객 |
| F-10 | 대환 가심사 (Step 2) | **판단**(가심사) | 적격 필터 탈락 시 사유를 모국어로 안내하고 제안하지 않음 | 승인이 아닌 회부(`REFER_TO_JB_ENGINE`)만 |
| F-11 | 전북은행 심사엔진 회부 | 후속액션 | 회부 실패 시 접수번호 미발급 | 최종 승인은 전북은행 심사엔진의 배타적 권한 |
| F-12 | 감사로그 | 검증(설명가능성) | 전 게이트 결정 사유코드 적재 | 접근통제 하 보존(분쟁 방어) |
| F-13 | STR 후보 큐 | 후속액션(AML) | 점수 70 이상 자동 적재·보류 | 보고 판단은 준법감시 |

#### (나) 유학생 기능 (v2 신규)

| ID | 기능 | 구현 등급 | 동작과 책임 경계 |
|---|---|---|---|
| **FR-S1** | 등록금 위임 송금 | 실동작 | 대학(화이트리스트) 대상 외화 등록금을 환율 조건 충족 시 자동 송금. 기존 3중 게이트를 `fx_pair="KRW/CNY"`로 그대로 재사용하며, 통지는 중국어로 제공 |
| **FR-S2** | 한도해제 코치 | 실동작(read-only) | 한도계좌에서 정식계좌로의 승급 조건(연속 급여 3개월 이상)을 결정적으로 판정·안내. 자금 이동 없음 |

| **FR-S4** | 졸업 전환 가심사 | 인상(재사용 준비) | 졸업과 취업비자 전환 시 대환 가심사로 즉시 연결. 기존 `/refi/*` 재사용. 원화 절약 헤드라인은 미노출(고정 246만 원과의 충돌 회피), 적격 여부와 가등급만 표시 |

> **v2 판정 규칙(결정적).** FR-S2 한도해제는 `activation.evaluate_limit_release`가 판정한다. `is_limited_account=true`이고 `salary_months_consecutive ≥ 3`이면 `eligible`, 미달이면 `in_progress`(남은 개월 안내), 비한도계좌면 `full_account`이다. FR-S1 등록금 송금은 Gate A 위임장 active와 수취인 in-scope, Gate B `KRW/CNY` 7일 평균 대비 1% 이상 유리와 한도 내, Gate C 대학 화이트리스트 AML PASS를 모두 충족할 때 `executed`로 처리한다.

> **구현(MCP 도구 및 엔드포인트).** F-04 `validate_mandate`, F-05 `check_limits`와 `evaluate_fx`, F-06 `screen_beneficiary_aml`, F-07 `execute_remittance`, F-10 `get_refi_offer`, F-11 `refer_to_jb_engine`. 위임장 발급은 `models.Mandate`, 모의 코어는 SQLite 12테이블. v2 신규 엔드포인트 3종은 §3-5 추적성 매트릭스 참조.

### 3-2. 위임장(Mandate) 스키마: 전자금융거래법 거래지시 3종

전자금융거래법상 반복 거래지시 요건을 위임장 한 건에 강제로 탑재한다. ① 전자서명 `esign.esign_hash`(위임 내용 SHA256 해시로 변조 방지, 모국어 재확인). ② 건별 통지 `notification.per_execution`. ③ 100% 철회권 `revocation.revocable`. 범위 필드로 `beneficiary_whitelist`(수취인 화이트리스트), `limits`(1회·월 한도), `fx_condition`(7일 평균 대비 1% 이상), `valid_until`(유효기간), `on_exception=hold_and_ask`(예외 시 보류)를 규정한다. 근로자와 유학생은 동일 스키마를 공유하며, 유학생 등록금 위임장은 수취인을 대학으로, `currency_target`을 `CNY`로 설정한다.

## 4. 주요 기능 흐름도 (Flow)

> 그림: `docs/diagrams.md` §1(유스케이스), §3(Step 1 시퀀스), §4(Step 2 시퀀스), §5(유학생 등록금), §6(가계좌 활성화)와 `docs/usecase.puml`(정식 UML)을 캡처해 삽입한다.

**근로자 데모 시나리오(한 흐름):**

1. **자동 송금**: (베트남어) "월급 오면 엄마한테, 환율 좋을 때만, 200만 원까지" → 위임장 발급(모국어 재확인과 전자서명) → 급여 감지 → FX +1.82% 충족 → 고객 개입 없이 자동 송금과 베트남어 통지.
2. **자동 보류**: 새벽 신규 수취인 480만 원 → AML 점수 75로 자동 보류와 "정말 보내려는 분이 맞습니까?" 모국어 질문 → STR 후보 큐 등록.
3. **사기 차단**: "OTP와 신분증을 보내면 대출" 수취인 → 블랙리스트 기반 즉시 차단과 "비밀번호와 OTP를 주지 마세요" 안내.
4. **대환 가심사(핵심 수익원)**: 급여와 사채 분석 → "연 246만 원을 절약할 수 있습니다" → 절약액과 월상환액과 연체 불이익을 동일 비중 모국어로 고지(금소법 모의) → [전북은행에 신청] → "최종 승인은 전북은행 심사".

**유학생 데모 시나리오(같은 엔진, 두 입구):**

1. **등록금 자동 송금**: (중국어) "환율 좋을 때 등록금 보내기" → 대학 화이트리스트 수취인 → 동일 3중 게이트(KRW/CNY +1.846%) 통과 → 등록금 송금 실행과 중국어·한국어 통지.
2. **한도해제 코치**: 가계좌에서 시작한 한도계좌가 연속 급여 3개월을 충족 → 정식계좌 전환 가능 안내(자금 이동 없음).
3. **재학 중 신용 형성**: 연속 급여와 정시 거래를 신용 이력 스냅샷으로 제시.
4. **졸업 전환 가심사**: 졸업과 취업비자 전환 시 기존 대환 가심사 경로로 연결(원화 절약 헤드라인 미노출, 적격과 가등급만 표시).

**Agent 루프:** 큰 루프는 **판단**(Gate A·B·C) → **행동**(실행·보류·알림) → **검증·개선**(사후 성과를 반영한 위임 임계값 조정)으로 구성한다. 세부 루프는 **수집 → 검색 → 판단 → 생성 → 검증 → 후속액션**으로 진행한다.

| 신용정보법 | 급여와 부채로 가심사 | 동의, 목적 외 금지, 가명처리(확장 마이데이터) | 동의 범위 내, 외부 LLM에 식별정보 비전달(토큰화) |
| 특정금융정보법 | 외국인과 자동화와 송금 | CDD/EDD, STR, CTR | Gate C AML 룰엔진, 점수 70 이상 STR 후보 큐 |
| 금융소비자보호법 | 자동 대환 제안은 권유 | 설명의무, 적합성, 부당권유 금지 | 절약액과 불이익을 동일 비중 모국어로 고지, 최종 승인은 전북은행 |

### 6-2. 오픈소스·외부데이터 출처·라이선스 (평가 3.5)

전부 허용형(permissive) 라이선스이다. Python(PSF), FastAPI(MIT), Uvicorn(BSD-3), Pydantic(MIT), SQLite(Public Domain), MCP SDK(MIT), React/Vite(MIT). LLM은 상용 API(식별정보 비전달, 학습 미사용 약관)를 사용한다. 데이터는 전부 모의·합성이므로 개인정보와 저작권 리스크가 없다. 환율은 모의 시드이며, 실 연동 시 한국은행 ECOS OpenAPI를 사용한다. 블랙리스트는 공개 보도 사실에서 패턴만 추출한다.
