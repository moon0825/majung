## Q1. "LLM이 의사결정하면 환각으로 오송금이 날 수 있지 않습니까?"

**핵심 포인트:** LLM은 실행 경로 밖에 있다.

**30초 구두 답변:**

"마중의 송금 실행 경로에 LLM은 없습니다. 채팅 문장에서 금액·수취인 파라미터를 추출할 때만 씁니다. 그 결과는 반드시 3중 결정적 게이트를 통과해야 실행됩니다. Gate A는 위임장 일치를 검증하고, Gate B는 한도·FX 조건을 점검하며, Gate C는 AML 리스크 스코어를 판정합니다. 환각이 발생해도 세 게이트 중 하나가 차단합니다. 이 구조는 코드베이스 동결 구역으로, 운영 중 변경이 불가합니다."

**예비 반론 대응:**

- "그래도 파싱 단계에서 틀리면?" → 파싱 결과는 위임장(수취인·한도·FX 조건)과 교차 검증합니다. 위임장에 없는 수취인은 Gate A에서 즉시 차단합니다.
- "프롬프트 인젝션은?" → LLM에 전달하는 컨텍스트는 채팅 입력문 하나뿐이며, 파싱 결과가 구조화 스키마를 벗어나면 전체 트랜잭션을 폐기합니다.

## Q2. "AML·KYC 규제 준수를 어떻게 보장합니까?"

**핵심 포인트:** 특금법 의무를 결정적 코드로 구현, 동결 임계치.

**30초 구두 답변:**

"Gate C에서 리스크 스코어 40 이상이면 자동 보류, 70 이상이면 즉시 차단합니다. 의심거래는 STR 대기열에 자동 적재되어 KoFIU 보고 프로세스로 넘어갑니다. 특금법의 CDD·CTR·STR 세 가지 의무를 결정적 코드로 구현했으며, 40·70 임계치는 동결 구역으로 운영 중 임의 변경이 불가합니다. 신규 계좌 1일 30만 원 한도 규제는 금융위 고시 기준 그대로 반영했습니다."

**예비 반론 대응:**

- "컴플라이언스 담당자가 임계치를 바꿔야 할 때는?" → 동결 구역 변경은 코드 리뷰와 테스트 통과, 은행 준법감시팀 승인을 거치는 프로세스로 분리됩니다. 즉흥 변경이 원천 차단됩니다.
- "STR 보고까지 자동화하려는 건가?" → 보고 결정은 사람이 합니다. 마중은 대기열 적재까지만 자동화하고, 최종 보고 여부는 담당자가 판단합니다.

## Q3. "연 246만 원 절약이라는 수치 근거는 무엇입니까?"

**핵심 포인트:** 출처 있는 외부 수치 두 개의 산술 결과, 자체 추정 명시.

**30초 구두 답변:**

"사채 금리 연 30%는 서울경제TV 2026년 3월 보도 기준이고, 전북은행 신용대출 금리 연 13.59%는 공시 기준입니다. 차이 16.41%포인트에 대환 원금 1,500만 원을 곱하면 연 246.15만 원입니다. 시장 수치 두 개의 산술 결과이므로 별도 조사가 필요하지 않습니다. 실제 대환액은 개인마다 다르며, 발표 자료에 '자체 추정(가정 명시)'으로 표기했습니다."

**예비 반론 대응:**

- "사채 금리 30%가 실제로 그렇게 높냐?" → 경향신문 2025.11 보도에 따르면 부산 사채조직이 외국인 9,120명에게 연 154%를 적용한 사례도 있습니다. 30%는 보수적 수치입니다.
- "전체 시장으로 확장하면?" → 대환 가능 대상 6.6만 명 기준 연 이자 절감 총액 추정치는 docs/출처맵.md §B에 있습니다.

## Q4. "카카오·토스가 똑같이 만들면 끝 아닙니까? 경쟁우위가 있습니까?"

**핵심 포인트:** 인프라 우위는 복제 불가, 마중은 활성화 레이어.

**30초 구두 답변:**

"전북은행은 이미 외국인 신용대출 시장 점유율 72%, 누적 24만 명, 대출 잔액 1조 원이라는 인프라를 보유합니다. 카카오와 토스는 인터페이스는 만들 수 있지만, 외국인 신용 이력이 없는 고객에게 실제 대출을 실행할 역량이 없습니다. 마중의 핵심 가치는 채팅 UI가 아니라, 이 72% 점유율 기반 고객을 송금 자동화와 사채 대환으로 이어주는 전환 구조입니다."

**예비 반론 대응:**

- "한패스가 이미 비슷한 것 아닌가?" → 한패스는 송금 레일 사업자(57만 명)이고 JB의 2대 주주(15%)입니다. 마중은 한패스 레일 위에서 동작하는 것을 전제하며, 경쟁이 아닌 협력 관계입니다.
- "전북은행만 할 수 있는 이유가 있나?" → 외국인 신용대출 72% 점유율과 JB금융그룹의 광주은행·JB우리캐피탈 계열사 시너지가 빠른 확장을 가능하게 합니다.

## Q5. "고객 개인정보(여권·계좌·송금 내역)를 LLM에 넘기면 보안 문제 아닙니까?"

**핵심 포인트:** LLM에는 채팅 텍스트만 전달, 민감 식별자 미전달.

**30초 구두 답변:**

"LLM에 전달하는 것은 채팅 입력 문장뿐입니다. 계좌번호·여권번호·생체인증 데이터는 LLM을 거치지 않습니다. 파싱된 결과(금액·수취인 코드)는 즉시 게이트 검증에 투입되고 원본 메시지는 폐기합니다. 실제 은행 배포 시에는 온프레미스 LLM 또는 API 레벨 데이터 마스킹이 전제되며, 이 부분은 전북은행 정보보호팀의 정책 결정 영역입니다."

**예비 반론 대응:**

- "데모에서는 실제로 API 키를 쓰지 않나?" → 데모는 완전 Mock 환경입니다. 외부 LLM API 호출 없이 미리 정의된 파싱 결과를 반환합니다.
- "MCP로 외부 AI가 계좌를 조회하면?" → MCP 연동 여부는 전적으로 전북은행이 통제합니다. read-only 조회 권한만 부여되며, 실행 권한은 마중 게이트가 독점합니다.
