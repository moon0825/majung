# 마중 본선 데모 빌드 표준 (jb-demo-build)

이 스킬은 jb-demo-engineer가 사용한다. 목적은 분명하다. 일주일 솔로의 데모 프로토타입을 심사위원 눈에 상용 제품으로 읽히는 완성도로 끌어올리되, 라이브가 무대에서 깨지지 않게 한다. 도달 목표는 JB 임원이 적용 가능성을 즉시 떠올리는 데모 질감이며, 진짜 운영 뱅킹 시스템 구축은 일주일 솔로의 범위 밖이다.

모든 작업의 제1전제는 하나다. 송금 절차에서 LLM을 배제하여 금융 사고를 원천 차단하는 3중 게이트 구조와 그 단일 실행 경로는 손대지 않는다. 새 작업은 표시 레이어 추가에 한정한다.

## 1. 코드베이스 지형 (무엇이 어디에 있나)

작업 대상은 `majung/`이며 백엔드와 프론트엔드로 나뉜다.

백엔드(`backend/app`, Python 3.11과 FastAPI, SQLite):
- `main.py`: 엔드포인트 13종(`/health`, `/chat`, `/remittance/execute`, `/fx/{base}/{quote}`, `/refi/prescreen`, `/refi/refer`, `/mandate/issue`, `/mandate/{id}/sign`, `/mandate/{id}/revoke`, `/audit/{user_id}`, `/str-queue`, `/salary/deposit`, `/notifications/{user_id}`).
- `orchestrator.py`: 3중 게이트(Gate A·B·C)를 순서대로 강제하고 LLM의 실행을 차단하는 단일 실행 경로. **동결 구역.**
- `rules.py`: FX, AML, 대환의 결정적 룰 엔진. **동결 구역.**
- `mcp_tools.py`: MCP 도구 13종. `models.py`: 위임장 등 Pydantic 스키마. `llm.py`: 의도 파싱 전용으로 실행 권한 없음. `db.py`와 `schema.sql`: SQLite 연결과 DDL. `seed.py`: 데모 시드.
- `tests/`: `test_gates.py`와 `test_new_endpoints.py`, 합계 20개를 유지한다.

프론트엔드(`frontend/src`, React와 Vite):
- `App.jsx`: 화면 오케스트레이션, 헬스체크 분기, 액션 5단계, 촬영 단축키. 동결 계약이 가장 많이 모인 파일이라 표시 레이어만 다룬다.
- `api.js`: 백엔드 호출 단일 창구. **동결 구역**(throw 금지 계약).
- `demoData.js`: 시드 상수와 오프라인 미리보기 목 데이터, 사업가치 콘솔 정적값 `BIZ`.
- `format.js`, `styles.css`, `main.jsx`.
- `components/`: `AdminDashboard.jsx`, `BusinessValuePanel.jsx`, `Cards.jsx`, `ControlPanel.jsx`, `CustomerChat.jsx`, `EmptyState.jsx`, `PhoneFrame.jsx`. 화면 보강은 대부분 여기에서 한다.

## 2. 목업과 실동작의 경계 (이식 아키텍처)

데모는 백엔드 생사에 따라 두 경로로 갈린다. 이 분기를 흐리면 라이브가 깨진다.

- 실 경로: `App.jsx`의 `healthRef.current`가 참이면 `api.js`를 거쳐 실제 백엔드 응답을 쓴다.
- 폴백 경로: 백엔드가 죽어 있으면 `demoData.js`의 `MOCK`을 쓴다. 백엔드가 살아 있으면 목은 절대 쓰지 않는다.

이식의 핵심 규칙은 다음과 같다.

1. `MOCK`의 응답 형태는 백엔드 응답 스키마와 글자 단위로 같게 유지한다. 게이트 배열(A·B·C), `status`, `message_ko`와 `message_local`, 대환 응답 필드까지 동일해야 폴백이 실 화면과 구분되지 않는다.
2. 화면에 박힌 정적 표시값은 백엔드 응답으로 옮길 수 있는 자리로 본다. 다만 옮긴 뒤에도 오프라인 폴백 값은 남겨 두어 백엔드가 없을 때 화면이 비지 않게 한다.
3. `BIZ`의 거시 수치(대환 전환율, 대환잔액, 연간 이자, 1인당 절약, 누적 차주, 점유율)는 현재 세션 행동과 무관한 정적값이다. 향후 집계 API로 교체할 자리로 표시하되, 시연 중 일관성을 위해 지금은 정적으로 둔다.
4. `api.js`의 반환 계약 `{ ok, status, data, error }`와 throw 금지 원칙은 바꾸지 않는다. 새 엔드포인트를 붙여도 같은 계약으로 감싼다.
5. 송금 실행 경로(orchestrator.py와 rules.py)의 판정 로직은 어떤 이유로도 수정하지 않는다. 실 데이터 연동은 `seed.py`와 데이터 소스 교체로 풀고, 판정 코드는 그대로 둔다.

이 경계를 지키면 데모는 백엔드를 실제 소스로 갈아 끼워도 표시 레이어를 다시 손볼 필요가 없다.

## 2-1. 다국어는 인상까지만 (한국어 병기 유지)

데모의 다국어는 완전 번역이나 언어별 빌드가 아니다. 목표는 주고객 나라에서 그대로 쓰겠다는 인상을 만드는 한 컷이다.

- 범위: 전체 화면을 모국어로 완역하지 않는다. 언어 토글 하나나 핵심 한두 화면(위임장 확인, 송금 통지)만 모국어로 보여 인상을 만든다.
- 한국어 병기: 모국어 문구 옆에 한국어를 항상 같이 둔다. 심사위원과 JB 임원이 읽어야 하고, 브라보코리아 리뷰의 "동의 문서가 전부 한국어"라는 불만을 모국어 병기로 해소하는 장면이 핵심 효과다.
- 구현: 다버전 빌드를 만들지 않는다. 표시 문자열 한 벌에 모국어와 한국어를 함께 담고 토글로 강조만 바꾼다. 게이트 판정 메시지의 `message_ko`와 `message_local`이 이미 이 구조이므로 그 계약을 그대로 따른다.
- 금지: 언어를 늘린다고 핸들러나 라우팅을 분기하지 않는다. 새 언어는 표시 데이터 추가일 뿐 실행 경로 변경이 아니다.

## 3. 동결 구역 (절대 불변, 표시 레이어만 변경)

11번 상용화 계획과 OVERNIGHT_ROUTINE이 합의한 동결선이다. 아래를 건드리면 라이브가 깨지고 3종 산출물의 일관성이 무너진다.

- 게이트 판정: Gate A·B·C의 순서와 단일 실행 경로.
- 동시성 잠금: `lock`과 `unlock`, `busyRef`. 연타와 동시 조작에 따른 중복 실행을 막는 장치다.
- 오프라인 폴백: `healthRef` 분기. 백엔드 생사에 따른 실 경로와 목 경로의 갈림.
- 연출 타이밍: `paced`. 감지에서 판정, 실행으로 이어지는 순서가 화면에서 읽히도록 보장하는 최소 대기.
- 통지 메아리 억제: `suppressNotifs`. 방금 누른 버튼이 만든 중복 알림 카드를 막는다.
- `api.js`의 throw 금지 계약: 데모 중 화면이 죽지 않게 하는 마지막 안전장치.
- 교차 고정 수치(4절 참조).

이 영역에 변경이 필요해 보이면 멈추고 리더에게 보고한다. 우회로는 표시 레이어 추가뿐이다.

## 3-1. 동결-안전 확장 패턴 (고치지 않고 더하기)

전제는 orchestrator.py와 rules.py 무수정이다(3절). 그런데도 신규 기능이 필요하면 동결 함수를 고치지 않고 다음 네 가지 우회로만 쓴다.

1. 신규 얇은 엔드포인트는 동결 함수의 공개 키워드 인자를 활용한다. 예를 들어 다른 통화 송금은 `run_remittance`를 고치지 않고 공개 키워드 인자로 호출한다(`run_remittance(..., fx_pair="KRW/CNY")`). 동결 함수가 이미 `fx_pair`를 키워드로 받으므로 호출부만 새로 둔다.
2. 둘째 페르소나는 `seed.py`에 행을 추가해 만든다. 기존 페르소나 행과 기존 테스트는 건드리지 않고 격리한다. 새 사용자, 새 위임장, 새 수취인을 추가만 하고 기존 시드 값은 그대로 둔다.
3. 페르소나 전환은 기존 핸들러를 매개변수화하지 않고 별도 surface를 추가해 푼다. 기존 핸들러에 분기 인자를 끼우면 동결 계약과 회귀가 흔들리므로, 새 진입 화면이나 새 라우트를 하나 더 두는 쪽이 안전하다.
4. 신규 룰은 `rules.py` 안에 넣지 않고 새 모듈에 만든다. FX, AML, 대환 룰 엔진은 동결이므로, 추가 판정이 필요하면 별도 모듈로 분리하고 동결 룰의 출력을 입력으로만 받는다.

공통 원칙은 확장을 추가로만 한다는 것이다. 동결 파일의 한 줄도 바꾸지 않는다. 이 패턴이 PR 단위로 지켜졌는지는 `jb-cloud-ops`의 동결 diff 공백 규칙이 기계적으로 증명한다.

## 4. 교차 고정 수치 (어디서나 토씨까지 동일)

데모, 대본, 슬라이드, 문서가 같은 숫자를 써야 채점의 3종 일관성이 선다. 빈칸으로 두지 않는다.

- 환율 예시: 7일 평균 18.12 대비 현재 18.45, 1.82% 유리.
- 사채 대환: 연 13.59% 적용, 연 246만 원 절감(대중 카피에서는 "연 약 250만 원" 라운드 허용).
- AML 스코어 임계치: 보류 40, 차단 70.
- 전북은행 외국인 신용대출: 점유율 72%, 누적 24만 명, 그룹 잔액 1조 돌파, 연말 목표 1.3~1.5조.
- 시장: 체류 외국인 273만, E-9 30만, E-9 소득의 56.5% 송금, 5년 송금 25조.
- 사채 실태: 1,500만~2,000만 원, 연 20~40%.
- 한패스: 외국인 57만, 연 1.4조 송금, JB 2대 주주 15%, 2026년 3월 코스닥 상장.
- 토스: 외국인 46만, 5개 언어.

그 밖의 수치는 `CLAUDE.md` 핵심 수치 치트시트와 03 문서에서 가져온다.

## 5. 빌드 검증 (매 작업 후 회귀)

표시 레이어를 바꿀 때마다 다음을 통과시킨 뒤에야 완료로 본다.

- 프론트 빌드: `cd frontend && npx vite build --outDir /tmp/majung-dist --emptyOutDir`. 윈도우 `dist` 권한 문제를 피하려 샌드박스 경로로 컴파일만 검증한다.
- 백엔드 테스트: `cd backend && pytest -q`. 20개를 유지한다. 하나라도 줄면 변경을 폐기한다.
- 신규 테스트는 추가만 한다. 기존 20개는 회귀 기준선이므로 그대로 통과해야 하고, 신규 기능에는 테스트를 새로 더한다. 합계는 늘 수 있어도 기존 20개가 줄거나 변형되어 통과하면 안 된다. 기존 회귀를 고쳐 맞추지 않는다.
- 기준점 회귀: 6/27 정상 동작 캡처와 5단계 e2e 시나리오(자동 송금, 보류와 STR, 사기 차단, 대환 가심사, 위임 철회)를 대조한다.
- 실패 시 처리: 빌드나 테스트가 깨지면 그 변경을 되돌리고 다음 항목으로 넘긴다. 절반만 든 변경을 남기지 않는다.

## 6. 배포 점검 (심사위원이 링크로 바로 여는 상태)

목표는 심사위원이 링크 하나로 진입해 즉시 동작을 보는 상태다. 배포 대상과 범위는 본선 형식 발표 후 확정하므로, 발표 전에는 절차만 준비한다.

기동 순서:
1. 백엔드: `python -m app.seed`로 데이터를 시드한 뒤 `uvicorn app.main:app --reload --port 8000`으로 띄운다.
2. 헬스 확인: `/health`가 응답해 상단 표시등이 "API 연결됨" 녹색으로 바뀌는지 본다.
3. 프론트: 라이브는 `npm run dev`, 정적 배포는 `vite build` 산출물을 호스팅한다.

배포 환경에서 확정할 항목:
- 진입 경로: 정적 프론트와 백엔드 호스팅을 단일 링크 뒤에 둔다.
- 프록시와 BASE: 개발에서는 vite 프록시가 `/api`를 8000으로 넘긴다. 배포에서는 `/api` 프록시나 CORS, 그리고 호출 BASE 경로를 환경에 맞게 확정한다.
- 폴백 안전망: 백엔드가 응답하지 않아도 오프라인 미리보기로 화면은 살아 있다. 다만 심사 시점에는 백엔드 녹색등을 확인한 뒤 시작한다.
- 회귀: 배포본에서도 5단계 e2e를 한 번 돌려 기준점과 대조한다.

## 7. 라이브 시연 런북 (무대에서 깨지지 않게)

산출물은 `_workspace/05_demo_runbook.md`다. 다음을 빠짐없이 담는다.

사전 준비(발표 직전):
- `python -m app.seed`로 월 한도 소진과 위임 철회 잔존을 초기화한다. 재촬영 전 필수다.
- 백엔드 8000을 먼저 띄우고 녹색등을 확인한 뒤 첫 액션을 한다. 헬스체크는 8초 주기이므로 등이 켜질 때까지 기다린다.
- 발표 노트북을 전체화면으로, 해상도 1920으로 고정하고 줌을 100%로 둔다. 1180px 폭 의존 레이아웃을 피한다.
- 촬영 모드(F 키)는 좁은 화면에서 더 안정적이므로 상황에 맞게 켠다.

진행:
- 데모 단계는 숫자 단축키로 실행한다. 1은 송금 발화 스크립트, 2는 급여 입금과 자동 송금, 3은 신규 수취인 보류와 STR, 4는 사기 수취인 차단, 5는 대환 가심사다.
- 채팅 시연은 입력창에 포커스를 둔다. 입력창 포커스 상태에서는 숫자 단축키가 키를 가로채지 않는다.

5대 실패 모드 차단:
1. 월 한도 소진이나 위임 철회 잔존: 발표 직전 시드로 초기화.
2. 폭 의존 레이아웃 붕괴: 전체화면과 해상도 1920 고정.
3. 백엔드 미기동: 8000 선기동과 녹색등 확인.
4. 단축키 오조작: 채팅은 입력창 포커스로 키 가로채기 차단.
5. 헬스체크 지연: 녹색등 확인 후 첫 액션.

백업 영상 전환 기준:
- 녹색등이 합리적 대기 안에 켜지지 않거나, 단축키가 무반응이거나, 화면 붕괴 징후가 보이면 무음 mp4 백업본으로 즉시 전환한다.
- 라이브 복구는 무대 위에서 시도하지 않고 발표 흐름을 백업 영상으로 이어 간 뒤 무대 밖에서 처리한다.

## 8. 입력과 출력, 팀 통신

작업 시작 전 `D:/JB_Fin_AI/JB_AI Challenge/.claude/jb-learnings.md`를 읽고, 이미 기록된 데모와 완성도 패턴은 처음부터 피한다. 작업을 마치면 이번에 발견한 패턴을 한 줄 추가한다. 형식은 `날짜 · 분류 · 패턴 또는 교훈 · 처방`이며 분류는 데모를 우선 쓴다. 같은 실패 모드나 동결 구역 침범이 두 번 이상 관찰되면 본 스킬에 정식 규칙으로 승격을 제안한다. 이전 데모 산출물이 있으면 전체를 다시 짜지 않고 바뀐 부분만 회귀 검증한다.

## 10. 형제 스킬 크로스링크

- 완성도 게이트와 어체: `jb-craft-standard`, `fin-biz-tone`, 윤문이 깊어지면 `humanize-korean`, 구조와 출처는 `korean-proposal`.

- 서사와 채점 정합: 슬라이드 서사는 `jb-deck-strategy`, 채점과 일관성은 `jb-scoring-rubric`. 본 스킬은 jb-demo-engineer 에이전트가 사용한다.
- PR 검수와 클라우드·배포 운영: `jb-cloud-ops`. 본 스킬이 동결 구역과 표시 레이어 빌드 규칙을 정의하면, 그 규칙이 PR 단위로 지켜졌는지(브랜치 체크아웃, 동결 diff 공백, 회귀, 데모 e2e, 두 게이트, 보고)는 jb-cloud-ops가 검수한다. 푸시와 머지와 배포 통제도 그 스킬을 따른다.

이 문서 자체도 위 게이트 기준을 지킨다. AI 지문이나 직역투가 보이면 즉시 고친다.
