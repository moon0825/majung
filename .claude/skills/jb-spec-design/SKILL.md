# 마중 설계 문서 표준 (jb-spec-design)

이 스킬은 jb-spec-architect가 사용한다. 목적은 분명하다. 일주일 솔로의 실동작 데모를 심사위원과 JB 실무진이 정식 설계 문서로 읽는 수준으로 옮긴다. SRS는 4.3 명세서 구체성과 3.4 구성도 실현성을 받치고, UX 정보구조도는 2.2 As-Is 접점과 4.2 실동작 정직성을 받친다. 설계는 표시 레이어와 문서 레이어만 전제하며, 송금 실행 경로의 동결 구역은 바꾸지 않고 있는 그대로 기술한다.

near-miss 경계를 먼저 둔다. 문장 윤문과 어체 교정만 필요하면 fin-biz-tone, AI 티 제거만 필요하면 jb-craft-standard, 데모 코드 빌드와 배포는 jb-demo-build, 채점과 일관성 검수는 jb-scoring-rubric이다. 본 스킬은 요구사항과 정보구조를 문서로 정식화하는 일에만 발동한다. 코드를 고치거나 다이어그램을 직접 렌더하는 일은 jb-demo-engineer 소관이다.

## 0. 두 산출물과 다이어그램 분담

산출물은 둘이다. SRS는 `_workspace/07_srs.md`, UX 정보구조도는 `_workspace/08_ux_ia.md`로 쓴다.

유스케이스와 시퀀스 다이어그램은 본 스킬에서 직접 그리지 않는다. 액터와 시나리오와 범위를 명세해 jb-demo-engineer에 넘기고, 코드 기준으로 렌더된 다이어그램 식별자를 받아 추적성 매트릭스에 연결한다. 왜 분담하는가. 코드를 아는 쪽이 코드 기준으로 그려야 실제 호출과 어긋난 환각 다이어그램을 피하고, 같은 그림을 양쪽이 따로 그리는 중복을 없앤다. 본 스킬은 다이어그램이 받칠 요구사항을 정의하고, 렌더는 위임한다.

## 1. SRS 골격

SRS는 일곱 묶음으로 짠다. 각 묶음은 무엇을 채울지와 왜 채우는지, 그리고 무엇이 미달인지를 함께 본다.

### 1-1. 개요(Introduction)

목적과 대상 독자(심사위원과 JB 실무진)와 서비스 정의(마중)와 문서 범위를 한 묶음으로 적는다. 왜. 읽는 사람이 첫 페이지에서 이 문서가 무엇을 보장하고 무엇을 보장하지 않는지 알게 한다. 미달 경계: 서비스 한 줄 소개만 있고 문서가 무엇을 다루는지 범위 진술이 없으면 개요가 아니다.

### 1-2. 범위(Scope)

포함과 제외를 가른다. 포함은 예선 MVP 실동작이다. 엔드포인트 13종과 5단계 e2e(자동 송금, 보류와 STR, 사기 차단, 대환 가심사, 위임 철회)가 여기다. 제외는 본선 로드맵이다. 음성과 17개국어와 실 API 연동이 여기다. 왜. 과대 약속을 막고 예선과 본선을 가르는 것 자체가 4.4 본선 고도화 기반과 8항 과대 약속 금지를 받친다. 미달 경계: 향후 자리를 현재 기능으로 적으면 범위가 아니라 과대 기술이며 4.2 정직성을 깬다.

### 1-3. 액터(Actors)

관여 주체를 모두 세운다. 고객(외국인 근로자, 응우옌 페르소나), 운영자(JB 준법감시와 여신심사 관리자), 외부 시스템(전북은행 심사엔진, 한패스 인가 레일, 환율 피드, 브라보코리아 전환 엔진)을 적는다. LLM은 의도 파싱 전용 액터로만 두고 실행 권한이 없음을 명시한다. 왜. LLM을 송금 실행 경로 밖에 둔다는 제1원칙과 3.1과 킬러질문 1의 책임소재 방어를 액터 정의에서부터 못박는다. 미달 경계: LLM을 실행 주체로 그리면 아키텍처 제1원칙 위반이다.

### 1-4. 기능요구(FR)

기능을 ID로 적는다(FR-1, FR-2 형식). 평가 6단계인 수집과 검색과 판단과 생성과 검증과 후속액션, 그리고 판단과 행동과 검증·개선을 기능 분류 라벨에 자구 그대로 보존한다. 다룰 기능은 위임장 발급과 서명과 철회, 위임 송금 3중 게이트, FX 룰 판단, AML 스코어 보류와 차단, 대환 가심사와 연계, 급여 입금 감지, 알림과 건별 통지, 감사와 STR 큐, 관리자 대시보드다. 각 FR에 액터와 트리거와 주경로와 예외, 그리고 실동작인지 인상인지를 함께 적는다. 왜. 6단계 용어를 라벨로 박아야 3.2 파이프라인 채점을 받고, 실동작·인상 표기가 추적성 매트릭스의 정직성을 받친다. 미달 경계: 기능 나열만 있고 트리거와 예외가 없으면 요구사항이 아니라 기능 목록이다.

### 1-5. 비기능요구(NFR)

품질 속성을 적는다. 규제(전자금융거래법의 전자서명과 건별 통지와 무조건 철회권, 외국환거래법 한도, 신용정보법, 자금세탁방지), 보안(거래 데이터 비식별 전송과 국내 리전), 설명가능성(결정적 룰과 근거 제시), 가용성(오프라인 폴백), 성능과 연출(paced 타이밍), 책임소재(LLM 실행 배제), 다국어(message_ko와 message_local 동시 제공)를 다룬다. 왜. 5.5 6대 리스크 표와 2.4 규제 인식과 직접 연결되며, 비기능요구가 비어 있으면 명세서가 기능 나열에 그친다. 미달 경계: 규제 법령명만 나열하고 어느 기능이 어떻게 충족하는지 연결이 없으면 인식 표시일 뿐 요구사항이 아니다.

### 1-6. 제약(Constraints)

바꿀 수 없는 조건을 적는다. 스택 고정(Python과 FastAPI와 SQLite와 React와 MCP SDK), 동결 구역(orchestrator.py와 rules.py와 api.js throw 금지 계약과 Gate A·B·C 순서), 솔로 일주일 범위, 양식 틀 유지와 PDF 변환이다. 왜. 설계가 실현 가능한 범위 안에 있음을 못박아 3.4 실현성을 받친다. 미달 경계: 동결 구역을 제약이 아니라 개선 대상으로 적으면 4절 위반이다.

### 1-7. 추적성 매트릭스

요구사항을 엔드포인트와 모듈과 UI와 다이어그램과 테스트까지 한 행으로 잇는다. 열은 요구사항 ID, 엔드포인트, 모듈, UI, 다이어그램, 테스트 여섯이다. 왜. 한 표가 4.1 일관성과 4.3 구체성과 4.2 실동작을 동시에 증명한다. 요구사항이 코드와 화면과 검증까지 닿아 있음을 한눈에 보인다.

| 요구사항 | 엔드포인트 | 모듈 | UI | 다이어그램 | 테스트 |
|---|---|---|---|---|---|
| FR-2 위임 송금 3중 게이트 | /remittance/execute | orchestrator.py, rules.py | CustomerChat, PhoneFrame | UC-2, SEQ-2 | test_gates.py |
| FR-3 위임장 발급·서명·철회 | /mandate/issue, /mandate/{id}/sign, /mandate/{id}/revoke | models.py, mcp_tools.py | ControlPanel | UC-3 | test_new_endpoints.py |

엔드포인트와 모듈과 테스트는 majung 코드에서 확인한 실재값으로 채운다. 다이어그램 열은 jb-demo-engineer가 부여한 식별자를 받아 채운다. 빈칸 열이 있으면 그 요구사항은 미구현이거나 인상이므로 그렇게 라벨한다. 빈칸을 채운 척하지 않는다. 미달 경계: 코드에 없는 엔드포인트나 테스트를 매트릭스에 적으면 환각이며 4.2 정직성을 깬다.

## 2. UX IA 3축 정의

UX 정보구조도는 세 축을 교차한다. 각 교차 상태(셀)에 실동작인지 인상인지를 반드시 라벨한다.

- 축1 페르소나: 고객(외국인 근로자, 응우옌)과 운영자(JB 준법과 여신 관리자) 둘을 기본으로 둔다. 사업주는 범위 밖이거나 인상으로 표기한다.
- 축2 계좌 생애주기: 신규 한도계좌, 급여 수령, 한도 해제와 주거래 전환, 위임 송금, 신용 형성, 사채 대환, 적금 연계 순서다. 시스템이 이 흐름을 강제하는 근거(신규 계좌 직후 20영업일 타행 개설 불가, 한도계좌 해제는 급여 50만 원 곱하기 3개월)를 함께 적는다.
- 축3 언어: 한국어와 모국어(베트남어) 둘이다. 모든 화면 상태에 message_ko와 message_local, 모국어 위임장, 모국어 자막이 어떻게 걸리는지 표기한다.

각 셀의 실동작·인상 라벨이 이 산출물의 핵심이다. 실동작 예: 위임 송금 자동 실행, AML 보류와 STR 후보, 사기 수취인 차단, 대환 가심사는 백엔드 게이트가 실제로 돈다. 인상 예: 사업가치 콘솔(BIZ)의 거시 집계 수치는 정적값이며 향후 집계 API 자리다. 적금 연계와 실 환율 API도 인상이다. 왜. 무엇이 진짜 도는지를 정직하게 가르는 것이 4.2 실동작 증명의 신뢰를 만든다. 인상을 실동작으로 칠하면 채점 신뢰가 통째로 무너진다.

표현은 화면 노드와 진입 경로와 상태 전이를 표나 트리로 그린다. 위계와 내비게이션 설계는 3절 규약으로 ui-ux-pro-max를 호출한다.

## 3. ui-ux-pro-max 호출 규약

- 호출: UX IA의 화면 위계와 정보 구조와 내비게이션과 상태 전이 설계에 ui-ux-pro-max 계열(ui-ux-pro-max:ui-ux-pro-max, ui-ux-pro-max:design-system)을 호출한다.
- 재구현 금지: 그 스킬을 베끼거나 다시 만들지 않는다. 위계 원칙과 구조 패턴 같은 결과만 흡수한다. 왜. 본 에이전트는 설계 문서를 만들지 컴포넌트를 구현하지 않는다.
- 범위 차단: 호출 결과가 코드 변경이나 새 컴포넌트 구현, 동결 구역을 제안하면 채택하지 않는다. 설계 산출물은 표시 레이어와 문서 레이어 안에만 머문다.
- 어체: ui-ux-pro-max가 만든 문구도 fin-biz-tone을 지킨다. 표준이 표준을 어기지 않는다.

## 4. 동결 구역 인지 (설계는 표시·문서 레이어만)

동결 구역은 jb-demo-build 3절과 같다. orchestrator.py와 rules.py, api.js의 throw 금지 계약, Gate A·B·C 순서와 단일 실행 경로, lock과 unlock과 busyRef, healthRef 분기, paced 타이밍, suppressNotifs, 교차 고정 수치다.

설계 산출물은 이 로직을 있는 그대로 기술하고 재설계를 제안하지 않는다. SRS의 기능요구와 비기능요구는 동결 동작을 요구사항 언어로 옮기는 작업이지 바꾸는 작업이 아니다. 예를 들어 3중 게이트는 FR로 "의도 파싱 다음 위임장 검증, 한도 룰, 수취인 화이트리스트 순서로 통과해야 실행한다"처럼 적되, 게이트 순서나 판정 로직을 손대는 제안은 넣지 않는다.

동결 로직 변경이 필요해 보이면 멈추고 리더에게 보고한다. 우회로는 문서 레이어 기술뿐이다.

## 5. 평가 용어와 고정 수치 보존

평가 핵심 용어를 자구 그대로 쓴다. 판단과 행동과 검증·개선, 그리고 수집과 검색과 판단과 생성과 검증과 후속액션이다.

고정 수치는 `_workspace/00_show_spec.md`와 jb-demo-build 4절을 토씨까지 동일하게 인용하고 빈칸으로 두지 않는다.

- 환율 예시: 7일 평균 18.12 대비 현재 18.45, 1.82% 유리.
- 사채 대환: 연 13.59% 적용, 연 246만 원 절감(대중 카피는 연 약 250만 원 라운드 허용).
- AML 스코어 임계치: 보류 40, 차단 70.
- 전북은행 외국인 신용대출: 점유율 72%, 누적 24만 명, 그룹 잔액 1조 돌파, 연말 목표 1.3~1.5조.
- 한패스: 외국인 57만, 연 1.4조 송금, JB 2대 주주 15%, 2026년 3월 코스닥 상장, 인가 레일.

show_spec과 이 목록이 어긋나면 show_spec을 우선으로 보고 총감독에게 정정을 요청한다.

## 6. 작성 절차

1. `_workspace/00_show_spec.md`와 `07_시스템설계.md`와 majung 코드를 읽는다.
2. 액터와 기능요구와 비기능요구와 제약을 코드 실재값 기준으로 추출한다.
3. 추적성 매트릭스를 채운다. 빈칸 열은 미구현이나 인상으로 라벨한다.
4. 다이어그램 명세(액터와 시나리오와 범위)를 jb-demo-engineer에 핸드오프하고 식별자를 회수해 매트릭스에 연결한다.
5. UX IA 3축을 작성하고 각 셀에 실동작·인상 라벨을 단다. 위계는 ui-ux-pro-max로 설계한다.
6. jb-polish 완성도 게이트를 통과한 뒤 jb-scoring-qa에 넘겨 사실과 일관성을 검수받는다.

## 7. 입력과 출력 데이터 계약

## 8. 학습 루프와 형제 스킬

작업을 시작하기 전 `.claude/jb-learnings.md`를 읽어 이미 기록된 설계 실수와 반복 패턴을 처음부터 피한다. 인상을 실동작으로 칠한 과거 지적이나 코드에 없는 항목을 매트릭스에 적은 사례를 다시 밟지 않는다. 작업을 끝낸 뒤에는 새로 잡은 패턴을 분류 `설계`로 한 줄씩 추가한다. 같은 패턴이 두 번 이상 잡히면 이 스킬의 표나 절로 승격을 제안한다. 이전 SRS나 UX IA가 있으면 전체를 다시 짜지 않고 바뀐 요구사항과 매트릭스 행만 갱신한다.

형제 스킬을 둔다. 어체 고정은 fin-biz-tone, AI 티 제거와 완성도 게이트는 jb-craft-standard와 jb-polish, 데모 빌드와 동결 구역과 코드 지형은 jb-demo-build, 채점과 일관성은 jb-scoring-rubric, 서사와 슬라이드는 jb-deck-strategy다. 전체 워크플로우는 jb-finals 오케스트레이터가 지휘한다. 이 스킬은 그 흐름의 앞단에서 요구사항과 정보구조를 정식 문서로 닫는 설계 게이트다.

이 문서 자체도 위 게이트 기준을 지킨다. AI 지문이나 직역투가 보이면 즉시 고치고 em dash는 쓰지 않는다.
